# Translation Expert Review Flow

トラウマケアサービスにおいて、誤訳された相談窓口情報や AI プロンプトはユーザーの安全を直接的に脅かしうる。本文書は、各言語の翻訳を本番投入する前に必ず通すべき専門家監修プロセスを定義する。

## なぜ専門家監修が必須なのか

- **危機リソース文言**: メンタルヘルス危機にあるユーザーが目にする、最も重要なテキスト。誤訳されたフレーズや不適切な相談窓口へ誘導すれば、人命に関わる。
- **AI プロンプト**: AI の挙動 (質問のトーン、回答の批判性、医療的助言の有無) を決定する。例えば「judge or criticize」を「批判する」と訳すと意味が狭くなる (英語では「評価・判定する」の含意もある)。
- **国・地域差**: 同じ言語でも国によって相談窓口が違う (例: スペイン本土とラ米諸国、サウジアラビアと UAE)。地域別の窓口リストが必要。

## 監修者の要件

各言語につき以下の条件を満たす人物が望ましい:

- 当該言語のネイティブまたは流暢な話者
- 臨床心理士・公認心理師相当の資格、もしくはトラウマケアに関する実務経験
- 該当国の自殺予防・危機介入リソースに精通
- 利益相反開示済み (相談窓口運営者からの依頼でないこと)

代替策として **国際的な NPO** (例: IFRC、MSF Mental Health) のスタッフでも可。

## レビュー手順

### Phase 1: 翻訳の用意 (開発側)

1. 機械翻訳 + 内部レビューで暫定翻訳を作成
2. 該当ファイルにマーカー (`reviewed: false`、コメント `// pending expert review`) を付与
3. 変更を PR で提出、`needs-expert-review/{lang}` ラベルを付ける

### Phase 2: 専門家への依頼

1. 翻訳対象を整理した依頼文書を作成 (本リポジトリの該当 commit を含める)
2. 監修者に文脈を説明:
   - サービスの対象ユーザー (mental health 課題、犯罪被害、紛争地帯避難民)
   - 文章が表示されるタイミング (危機検知時、AI 質問時、UI 上の各箇所)
   - 表示されるユーザーの想定状態 (動揺している、緊急性が高い等)
3. 機密保持契約 (NDA) を結ぶ

### Phase 3: 監修者によるレビュー

監修者は以下の観点で確認:

- **正確性**: 用語が原文の意味を正しく伝えているか
- **トーン**: トラウマインフォームドケアの原則に沿っているか (穏やか、判断しない、希望のある言葉)
- **文化的適合**: 当該文化圏で違和感のない表現か (例: 直接的すぎる/間接的すぎる)
- **法的・医療的境界**: 医療助言と誤解されない言い回しか
- **相談窓口の正確性**: 電話番号・URL が現在も有効か、提供時間帯は適切か

### Phase 4: 取り込み

1. 監修者からの修正案を反映
2. `reviewed: true` に変更、commit メッセージに監修者の名前 (合意があれば) と日付を記録
3. PR をマージ
4. `docs/i18n-review-flow.md` の「監修済み翻訳の履歴」セクションに追加

### Phase 5: 定期見直し (年 1 回)

- 相談窓口の電話番号・URL・運営状況の変化を確認
- 当該国の自殺予防戦略・新しい hotline の有無を確認
- 監修者にフォローアップを依頼

## 緊急修正の取扱い

「相談窓口の番号が間違っていた」など、緊急性の高い修正:

1. すぐに `reviewed: false` に戻し、暫定的に IASP の international fallback に置き換え
2. 国内専門家を再度確認、修正値を入手
3. ホットフィックスとして別 PR で出す
4. ロールバック手順は `docs/deployment/rollback.md` 参照

## クライアント側の表示要件

`reviewed: false` の翻訳は本番ビルドでは出力しない、もしくは UI で「Pending review (暫定翻訳です)」と表示することが望ましい。実装は将来の Phase で:

- bff の SSR で `reviewed` フラグを HTML data attribute として埋め込み
- フロントエンドが `reviewed=false` の領域に小さなマーカーを表示
- ユーザーがフィードバックできるリンクも提供

これにより、暫定翻訳が誤って本番投入された場合のリスクが下がる。

## Q&A

**Q: 暫定翻訳でもユーザーが見られるなら、どうやって安全性を担保するのか?**

`hotlines` 配列には **必ず IASP の international fallback (`reviewed: true`)** が含まれる。これにより、暫定翻訳の窓口情報が間違っていても、ユーザーは確実に有効な国際リストへたどり着ける。

**Q: 機械翻訳の品質が高ければ専門家監修不要では?**

NO。機械翻訳は文脈を扱えない。「I'm fine」が「大丈夫です」とも「私は元気です」とも訳せるが、トラウマケア文脈では前者が正しく、後者は不適切。さらに専門用語 (例: trigger warning、suicidal ideation) は機械翻訳では誤訳が多い。

**Q: 監修費用が高い場合は?**

オープンソースのコミュニティ監修も検討可能。ただし以下の品質保証が必要:
- 複数人 (3 人以上) のレビュー
- それぞれ独立にチェックし、意見が一致しない場合は議論
- 最終的に専門家 (有償でも 1 回) による spot check
